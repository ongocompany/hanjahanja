# WSD 합성 학습 데이터 — 프롬프트 설계

## 1. 현황 분석

### 학습 데이터 불균형 현황 (725개 동음이의어)
- 한자 매핑 2개+ 존재하는 실질적 동음이의어: **501개**
- 소수 의미 (샘플 < 30): **768개 sense** (보강 타겟)
  - 심한 불균형 (10:1+): 173개 단어
  - 중간 불균형 (3:1~10:1): 139개 단어
  - 양호 (3:1 이하): 189개 단어

### 대표적 불균형 예시
| 단어 | 다수 의미 (샘플) | 소수 의미 (샘플) | 비율 |
|------|-----------------|-----------------|------|
| 자신 | 自身 (2,125) | 自信 (34) | 62:1 |
| 시장 | 市場 (937) | 市長 (17) | 55:1 |
| 입장 | 立場 (594) | 入場 (4) | 148:1 |
| 감정 | 感情 (668) | 鑑定 (21) | 32:1 |
| 의사 | 醫師 (321) | 意思 (151), 議事 (4) | 80:1 |

### 결론
→ 모델이 다수 의미로 편향. 소수 의미 문장을 집중 생성해야 함.

## 2. 프롬프트 설계 원칙

### Gemini 조언 + 국제 연구 종합

1. **소수 의미 집중 (Hard Negative Mining)**
   - 모델이 이미 잘 맞추는 다수 의미는 생성 불필요
   - 소수 의미에 대해서만 문장 생성 → 불균형 해소

2. **뜻풀이 포함 (GlossBERT 접근)**
   - 한자만 주면 LLM이 의미를 혼동할 수 있음
   - 우리말샘 정의를 프롬프트에 포함 → 정확한 의미의 문장 생성

3. **대립쌍 생성 (Contrastive Pairs)**
   - 같은 단어의 서로 다른 의미를 한 번에 요청
   - "자신(自身)의 문장 3개 + 자신(自信)의 문장 3개" 식으로
   - LLM이 두 의미의 차이를 명확히 인식한 상태에서 생성

4. **문체 다양성**
   - 뉴스, 일상대화, SNS, 블로그, 교과서, 소설 등
   - 기존 코퍼스가 뉴스/문어체 편향이므로 구어체 비중 높이기

5. **Thinking OFF**
   - Gemini 2.5 Flash, thinking 비활성화
   - 문장 합성은 직관적 생성 태스크 → 추론 불필요
   - 비용: 거의 무료 수준

## 3. 프롬프트 구조

### 3.1 시스템 프롬프트 (System Prompt)

```
너는 한국어 자연어 처리(NLP) 모델 학습을 위한 '합성 데이터 생성 전문가'야.
너의 임무는 주어진 한자어 동음이의어가 포함된, 실생활에서 흔하게 쓰이는 자연스러운 문장을 생성하는 거야.

[핵심 규칙]
1. 각 의미별로 문맥만 읽어도 어떤 뜻인지 명확히 구별 가능해야 함
2. 문체를 다양하게 섞어: 뉴스 기사, 일상 대화, 블로그, SNS, 교과서, 소설, 업무 메일 등
3. 타겟 단어가 반드시 문장에 자연스럽게 포함되어야 함
4. 같은 패턴 반복 금지 — 다양한 주어, 서술어, 상황 사용
5. 각 문장은 15자 이상 100자 이하
6. 출력은 반드시 JSON 배열만 출력. 다른 설명, 인사말, 마크다운 절대 금지
7. JSON 키: "sentence" (한글 문장), "word" (타겟 단어), "sense" (의미 번호: A 또는 B)
```

## 4. 생성 전략

### 4.1 단어별 생성량

| 불균형 수준 | 소수 의미 샘플 | 소수 의미 생성량 | 다수 의미 생성량 |
|------------|--------------|----------------|----------------|
| 극심 (50:1+) | 0-10개 | 50개 | 10개 |
| 심함 (10:1~50:1) | 10-30개 | 40개 | 10개 |
| 중간 (3:1~10:1) | 30-100개 | 30개 | 5개 |
| 양호 (3:1 이하) | 100+개 | 생성 안함 | 생성 안함 |

**예상 총량**: ~312개 단어 × 평균 50문장 ≈ **15,600문장**

### 4.2 비용 예상 (Gemini 2.5 Flash, thinking OFF)

| 항목 | 수치 |
|------|------|
| API 호출 수 | ~312회 (단어당 1회) |
| 입력 토큰/회 | ~800 (시스템+퓨샷+요청) |
| 출력 토큰/회 | ~2,000 (50문장 JSON) |
| 총 입력 | ~250K 토큰 |
| 총 출력 | ~624K 토큰 |
| Gemini 2.5 Flash 가격 | 입력 $0.15/M, 출력 $0.60/M |
| **예상 비용** | **~$0.41** (약 600원) |

### 4.3 품질 검증

1. **기본 검증** (자동):
   - 타겟 단어가 문장에 포함되는지
   - 문장 길이 15-100자
   - JSON 파싱 가능한지

2. **의미 검증** (LLM 교차 검증):
   - 생성된 문장을 다시 LLM에게 "이 문장에서 '{word}'는 A와 B 중 어떤 의미?" 질문
   - 원래 의도한 의미와 불일치하면 제거
   - 비용: 생성의 ~20% 추가 ($0.08)

3. **샘플 사람 검수**:
   - 드라이런 결과 10개 단어를 직접 확인
